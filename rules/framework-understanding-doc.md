# 规则：自动生成《框架理解文档》

## 1. 目标

当 AI 需要分析一个现有框架、代码库或项目骨架时，必须自动生成一份**高密度、强解释性、面向工程理解**的《框架理解文档》。

这份文档的核心目标不是“简单总结”，而是：

1. 帮助阅读者快速建立整个框架的全局认知
2. 帮助阅读者理解每个目录、每个核心文件、每个类/模块的职责
3. 帮助阅读者理解模块之间如何协作
4. 帮助阅读者理解系统从启动到运行再到销毁的完整流程
5. 帮助阅读者快速找到扩展点、修改点、排查点
6. 帮助阅读者在最短时间内具备“能读、能改、能扩展”的能力

---

## 2. 基本原则

生成《框架理解文档》时，必须遵守以下原则：

### 2.1 不要只停留在“这是什么”
必须继续解释：

- 它为什么存在
- 它解决什么问题
- 它和谁协作
- 它在整个架构里的位置是什么
- 如果没有它，系统会缺什么
- 它通常在什么时候被调用
- 它最容易被误解的点是什么

### 2.2 不要只按文件列表罗列
必须从以下多个维度同时组织内容：

- 架构维度
- 模块维度
- 目录维度
- 文件维度
- 生命周期维度
- 调用链维度
- 数据流维度
- 扩展点维度

### 2.3 不要泛泛而谈
必须尽量输出“工程级理解”：

- 文件职责要明确到位
- 类职责要明确到位
- 重要函数要说明作用
- 模块边界要清晰
- 依赖关系要清晰
- 控制流和数据流要清晰

### 2.4 面向初次接触该框架的人
文档必须假设读者：

- 第一次接触该框架
- 对代码结构不熟
- 需要快速上手
- 希望知道“先看什么，后看什么”
- 希望理解“这个文件到底是干嘛的”

### 2.5 解释必须“通俗 + 工程准确”
允许使用工程术语，但每次出现关键术语时，必须尽量解释清楚，不要只堆术语。

例如不能只说：

- Reactor
- EventLoop
- Dispatcher
- Context
- Adapter
- Facade
- Registry

而要进一步解释：

- 它在这个项目里具体扮演什么角色
- 它和传统定义相比，这里是否有特殊实现
- 它在本项目中是如何落地的

---

## 3. 输出文档的固定结构

每次生成《框架理解文档》时，必须尽量包含以下章节。除非项目极小，否则不得省略。

---

# 《框架理解文档》

## 0. 文档摘要

先用 5～15 条说明：

- 这个框架是干什么的
- 它主要解决什么问题
- 它的核心设计思想是什么
- 它最重要的几个模块是什么
- 阅读它时最先该抓住什么
- 它和常见同类框架相比最大的特点是什么

这一节要让读者在 1 分钟内建立总体印象。

---

## 1. 框架整体定位

这一节必须说明：

- 框架的主要用途
- 所属技术领域（例如网络库、Web 框架、游戏服务器框架、渲染框架、编译器前端、工具链等）
- 框架的边界在哪里
- 框架负责什么，不负责什么
- 框架最核心的抽象是什么
- 框架的运行模型是什么

应尽量回答以下问题：

- 这是一个什么层级的框架？
- 它是基础设施，还是业务承载层？
- 它偏通用能力，还是偏具体场景？
- 使用者通常如何接入它？

---

## 2. 目录结构总览

必须输出项目的目录树摘要，并逐层解释目录职责。

要求：

- 不能只贴目录树
- 每个重要目录都要解释“为什么存在”
- 要说明目录之间的边界
- 要说明哪些目录是核心运行时，哪些目录是工具层，哪些目录是示例层，哪些目录是测试层

示例说明维度：

- src/：核心源码
- include/：对外接口
- examples/：示例程序
- tests/：测试
- docs/：文档
- scripts/：脚本
- cmake/：构建辅助逻辑
- config/：配置
- internal/：内部实现细节
- platform/：平台适配层

---

## 3. 核心模块地图

必须将框架拆成几个核心模块，而不是直接陷入文件细节。

每个模块必须说明：

1. 模块名称
2. 模块职责
3. 为什么需要这个模块
4. 它依赖哪些模块
5. 哪些模块依赖它
6. 它暴露什么能力
7. 它隐藏什么复杂性
8. 它在系统中的地位（核心、支撑、工具、适配、胶水层）

对于每个模块，尽量额外补充：

- 典型入口类
- 关键数据结构
- 关键流程
- 易错点
- 阅读建议

---

## 4. 启动流程 / 生命周期分析

必须解释系统从“开始运行”到“进入稳定运行态”的过程。

必须覆盖：

- 程序入口在哪里
- 初始化顺序是什么
- 配置在哪里加载
- 核心对象在哪里创建
- 模块在哪里注册
- 主循环或事件循环如何启动
- 请求 / 连接 / 任务 /事件如何进入系统
- 系统关闭时如何清理资源

如果项目中存在以下内容，也要尽量分析：

- 单例初始化
- 全局注册表
- 工厂注册
- 插件加载
- 路由注册
- handler 注册
- coroutine / scheduler 初始化
- 线程池初始化
- 网络监听初始化

这一节必须让读者理解：

**程序到底是怎么“活起来”的。**

---

## 5. 主调用链分析

必须从“外部输入进入系统”开始，梳理至少一条主调用链。

例如：

- 一次网络请求如何流转
- 一条消息如何被解析、路由、处理、回包
- 一个任务如何被提交、调度、执行、完成
- 一个 UI 事件如何进入逻辑层
- 一个脚本调用如何进入引擎层

要求：

- 尽量按顺序列出关键步骤
- 每一步说明涉及哪些文件/类/函数
- 说明本步的输入、输出、职责变化
- 说明控制权在谁手里
- 说明数据在谁手里

如果系统有多条典型主链路，应至少列出 2～3 条。

---

## 6. 关键数据流分析

除了调用链，还必须分析数据流。

必须尽量说明：

- 配置数据如何进入系统
- 请求数据如何被解析
- 运行时状态如何保存
- 对象之间如何传递上下文
- 响应数据如何生成
- 错误如何传播
- 资源如何管理

重点关注：

- 所有权归属
- 生命周期边界
- 是否存在共享状态
- 是否存在线程间传递
- 是否存在异步恢复点
- 是否存在缓存层 / 中间层 / 队列层

---

## 7. 文件级职责拆解（重点）

这一节是整份文档的核心，必须尽可能详细。

对每个重要文件，必须使用如下模板分析：

### 文件：`路径/文件名`

#### 1）这个文件的核心职责
说明它“本质上是干什么的”。

#### 2）为什么这个文件必须存在
说明如果没有它，会导致什么问题，或者哪些职责会无处安放。

#### 3）它在系统中的位置
说明它属于哪一层：

- 接口层
- 核心调度层
- 数据结构层
- 工具层
- 平台适配层
- 业务承载层
- 桥接层
- 胶水层
- 启动层
- 测试层

#### 4）它依赖谁
列出它主要依赖的文件/类/模块，并说明依赖原因。

#### 5）谁依赖它
说明它主要被谁使用，它对外提供什么价值。

#### 6）文件中的关键类 / 结构体 / 函数
逐个解释其职责，不要只列名字。

#### 7）阅读这个文件时应该重点看什么
指出阅读抓手，例如：

- 先看类成员
- 再看构造过程
- 再看主函数
- 再看回调注册点
- 再看状态切换逻辑

#### 8）这个文件最容易让人误解的地方
例如：

- 名字看起来像管理器，实际只是薄封装
- 表面像数据结构，实际上承担调度职责
- 看起来是工具类，实际上决定主流程
- 接口很少，但地位很核心

#### 9）如果我要修改/扩展这个文件，应该注意什么
例如：

- 是否有线程安全问题
- 是否有生命周期问题
- 是否有回调悬空风险
- 是否有 ABI / 接口稳定性问题
- 是否影响上下游模块

---

## 8. 类级职责拆解（针对关键类）

对于关键类，必须额外做类级别解释。

模板如下：

### 类：`ClassName`

#### 1）类的定位
它是资源持有者、调度器、上下文对象、接口抽象、适配器、工厂、注册中心，还是其他角色？

#### 2）核心职责
只说“最本质”的职责，不要把所有事情都堆上去。

#### 3）关键成员变量的意义
解释每个重要成员变量的设计目的。

#### 4）关键方法职责
解释每个关键方法干什么、何时调用、由谁调用。

#### 5）对象生命周期
这个类是谁创建的、何时销毁、是否被共享、是否跨线程。

#### 6）协作关系
这个类和哪些类一起工作？谁驱动谁？谁持有谁？

#### 7）设计意图
为什么会设计成这样？为什么不是另一种写法？

#### 8）扩展点 / 重构点
如果要扩展功能，最佳切入点在哪？
如果要重构，风险最大的是哪部分？

---

## 9. 依赖关系与分层说明

必须梳理框架中的依赖方向。

要回答：

- 哪些层可以依赖哪些层
- 哪些层不能反向依赖
- 是否存在循环依赖
- 是否存在“看起来是底层，实际偷偷依赖高层”的问题
- 哪些文件是稳定抽象
- 哪些文件是变化集中区

应尽量明确：

- 对外 API 层
- 内部实现层
- 平台封装层
- 算法/数据结构层
- 运行时层
- 扩展层
- 测试辅助层

---

## 10. 扩展点地图

必须专门输出“扩展点”章节。

回答：

- 如果我想新增一个模块，应该从哪下手
- 如果我想新增一个 handler / route / component / subsystem，应该改哪里
- 如果我想替换底层实现，应该动哪里
- 如果我想插入日志、监控、埋点、profiling，应该插哪里
- 如果我想做热更新、插件化、脚本化，哪些位置最适合改造

扩展点必须尽量具体到文件和类。

---

## 11. 调试与排错入口

必须输出“出问题时先看哪里”。

例如：

- 启动失败先看哪里
- 请求没进来先看哪里
- 消息没分发先看哪里
- 回调没触发先看哪里
- 数据不同步先看哪里
- 崩溃先看哪里
- 死锁/阻塞/卡死先看哪里
- 内存泄漏先看哪里

这部分要帮助读者建立“排障路径”。

---

## 12. 推荐阅读顺序

必须给出“从哪里开始读最不容易迷路”。

例如：

### 第一轮：建立全局认知
- 先看入口文件
- 再看核心对象
- 再看主流程
- 再看关键抽象

### 第二轮：理解运行机制
- 看事件循环 / 调度器
- 看消息分发
- 看资源管理

### 第三轮：理解扩展方式
- 看接口层
- 看注册机制
- 看业务接入点

### 第四轮：深入底层细节
- 看性能关键点
- 看并发模型
- 看内存与生命周期设计

---

## 13. 框架设计优点 / 风险点

必须客观说明：

### 优点
- 分层是否清晰
- 抽象是否稳定
- 扩展性是否好
- 性能路径是否短
- 调试成本是否低

### 风险点
- 哪些地方耦合偏高
- 哪些地方理解门槛高
- 哪些地方未来容易膨胀
- 哪些地方可能成为性能瓶颈
- 哪些地方可能因为抽象不当导致维护困难

---

## 14. 最终总结

最后必须给出总结：

- 这个框架最该抓住的 3～7 个核心点
- 阅读者接下来最应该做什么
- 如果要快速上手，应该先尝试改哪几个文件
- 如果要真正掌握，下一步该深入哪些模块

---

## 4. 详细度要求

生成《框架理解文档》时，必须满足以下详细度要求：

### 4.1 文件级说明要足够细
不能只写：

- 负责网络
- 负责调度
- 负责配置

必须进一步写清：

- 具体负责什么边界
- 不负责什么
- 和谁配合
- 它在主流程里是什么位置
- 为什么职责放在这里最合理

### 4.2 对“核心文件”必须重点展开
以下类型文件通常属于核心文件，必须优先详细分析：

- main / app / bootstrap / startup
- EventLoop / Reactor / Scheduler / Dispatcher
- TcpConnection / Session / Channel / Poller / Acceptor
- Router / Handler / Registry / Factory
- Context / Request / Response
- Config / Options / Settings
- Runtime / Engine / Executor
- 模块注册中心
- 插件管理器
- 协程调度器
- 线程池 / 任务系统
- 内存池 / Buffer / Queue

### 4.3 对“看起来普通但实际关键”的文件要特别提醒
例如：

- 某个 util 文件实际承载全局约束
- 某个 small wrapper 实际决定资源生命周期
- 某个 base class 实际是系统扩展入口
- 某个 registry 文件实际是整个框架的拼装中心

---

## 5. 图示要求

如果输出格式允许，必须尽量附带以下图示之一或多个：

- 模块依赖图
- 启动流程图
- 请求调用链图
- 核心类关系图
- 生命周期图

优先使用：

- Mermaid
- 分层文字图
- 缩进树状结构
- ASCII 结构图

但图不能替代文字解释，图只是辅助。

---

## 6. 输出风格要求

文档风格必须满足：

- 中文输出
- 解释型写法，不要只列点
- 偏工程分析风格
- 允许长文，但必须层次清晰
- 小节标题明确
- 重点部分必须展开讲透
- 不要空话
- 不要套模板式废话
- 不要只有结论没有原因

---

## 7. 特殊要求：必须覆盖“每个文件的核心职责”

如果项目文件较多，至少必须做到：

1. 所有核心文件逐个分析
2. 非核心文件按目录或类别批量说明
3. 任何一个被点名的文件，都必须单独展开职责分析
4. 如果文件数量特别多，可以采用：
   - 核心文件详细拆解
   - 次级文件分组说明
   - 自动标记“建议后续再深入”的文件

但绝不允许直接跳过文件职责分析。

---

## 8. 文件职责分析的分组策略

当文件很多时，可按以下方式分组：

- 启动文件组
- 核心调度文件组
- 网络 I/O 文件组
- 数据结构与基础设施组
- 配置与构建组
- 测试与示例组
- 工具与辅助组
- 平台适配组

然后在组内继续拆到单文件。

---

## 9. 面向 AI 的执行步骤

当 AI 接到“分析框架/生成框架理解文档”任务时，应按如下步骤执行：

### 第一步：建立全局目录认知
先扫描目录结构，识别：

- 入口文件
- 核心源码目录
- API 暴露目录
- 示例目录
- 测试目录
- 构建目录
- 配置目录

### 第二步：识别核心模块
根据命名、依赖、入口关系，划分核心模块。

### 第三步：寻找主流程
优先找到：

- main
- app/bootstrap
- 初始化逻辑
- 注册逻辑
- 主循环
- 请求入口

### 第四步：识别关键抽象
例如：

- Loop
- Connection
- Context
- Handler
- Dispatcher
- Task
- Scheduler
- Registry
- Buffer

### 第五步：逐文件建立职责表
为每个重要文件建立：

- 文件名
- 所属模块
- 核心职责
- 上游
- 下游
- 关键类
- 关键函数
- 阅读优先级

### 第六步：生成最终文档
按本规则规定的结构输出完整理解文档。

---

## 10. 输出中的禁止事项

生成《框架理解文档》时，禁止：

### 10.1 只做表面摘要
不能只说“这是一个网络模块”“这是一个工具类”。

### 10.2 只贴目录树不解释
目录树必须配职责说明。

### 10.3 只贴类名函数名不解释
必须解释其工程意义。

### 10.4 忽略文件之间关系
必须说明依赖、调用、协作。

### 10.5 不讲运行流程
必须解释程序如何启动、如何流转。

### 10.6 不讲阅读顺序
必须告诉读者先看什么再看什么。

### 10.7 不讲修改风险
必须指出哪些文件改动风险高。

---

## 11. 质量自检清单

输出前，AI 必须自检：

- 是否已经说明框架的整体定位？
- 是否已经解释目录结构而不是只罗列？
- 是否已经划分核心模块？
- 是否已经说明启动流程？
- 是否已经分析主调用链？
- 是否已经分析关键数据流？
- 是否已经覆盖核心文件职责？
- 是否已经对关键类进行拆解？
- 是否已经说明扩展点？
- 是否已经说明调试入口？
- 是否已经给出推荐阅读顺序？
- 是否已经指出优点和风险点？
- 是否已经足够详细到让初学者快速上手？

如果上述任一项答案是否，则文档不能算完成。

---

## 12. 推荐输出文件名

当 AI 生成这类文档时，推荐使用以下文件名之一：

- `FRAMEWORK_UNDERSTANDING.md`
- `docs/framework-understanding.md`
- `docs/architecture/framework-guide.md`
- `docs/源码导读.md`
- `docs/框架理解文档.md`

---

## 13. 一句话触发词

当用户出现以下意图时，应自动应用本规则：

- 帮我分析这个框架
- 帮我看懂这个项目
- 帮我生成源码导读
- 帮我做框架理解文档
- 帮我解释整个项目结构
- 帮我拆解每个文件的职责
- 帮我快速理解这个仓库
- 帮我做架构阅读笔记

---

## 14. 最终要求

最终输出必须让读者达到以下效果：

- 知道框架是怎么跑起来的
- 知道每个核心模块是干什么的
- 知道每个重要文件为什么存在
- 知道该从哪开始读源码
- 知道以后该去哪改代码
- 知道出问题时先排查哪里

如果做不到这些，说明文档还不够好，还需要继续展开。
